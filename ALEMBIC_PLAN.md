# Plan: Managing ORM database schema (Alembic?)

## Recommendation (read this first)

**Don't adopt full Alembic yet.** Two facts about this codebase weaken the usual
case for it, and one narrow gap is the only real motivation:

- The relational DB is a **derived projection**, not the source of truth — the
  serialized protos in `ord-data` are
  ([orm/README.md](ord_schema/orm/README.md)). Recovery today is "rebuild from
  protos," so schema state is reproducible without a migration history.
- The ORM models are **generated from the proto descriptors** at import
  (`build_mappers()` in [mappers.py](ord_schema/orm/mappers.py) derives every
  table, column, and PG enum from the message definitions). Schema changes are
  driven by editing `.proto` + regenerating, not by hand-authoring SQLAlchemy
  models — which is exactly the workflow Alembic's autogenerate is weakest at.

The **one real gap**: `Base.metadata.create_all()` skips tables that already
exist, so *additive* changes (a new index or column) never reach a long-lived
database. The partial-index backfill that currently needs a manual
`CREATE INDEX CONCURRENTLY` ([orm/README.md](ord_schema/orm/README.md)) is the
canonical example.

So the honest verdict: do the **lightweight option (A)** now — it closes the gap
with far less machinery. Adopt **full Alembic (B)** only when one of these
triggers fires:

- A **long-lived production ORD database** (e.g. the public search instance)
  becomes large enough that rebuild-from-`ord-data` means unacceptable downtime.
- **Multiple independently-operated deployments** need reproducible, versioned
  upgrades you can't coordinate by rebuilding.
- Schema changes start including **destructive/transforming** steps (renames,
  backfills, type changes) that `create_all` fundamentally can't express.

Until then, Alembic adds a second source of schema truth (migration history vs.
proto-generated metadata) that must be kept in sync forever — real, permanent
overhead. The rest of this doc covers both options so the decision is concrete.

## Current state

- `prepare_database()` ([database.py:42](ord_schema/orm/database.py#L42)):
  `CREATE SCHEMA ord/rdkit`, `CREATE EXTENSION rdkit` (+ `tsm_system_rows`,
  `btree_gist` fallback), then `Base.metadata.create_all(engine)`.
- Models generated by `build_mappers()`; PG enums created from proto enum names
  ([mappers.py:178](ord_schema/orm/mappers.py#L178)).
- Partial indexes over unlinked rows defined as `Index(...)` with
  `postgresql_where` ([mappers.py](ord_schema/orm/mappers.py)).
- No `alembic_version` table, no migrations directory.
- Tests spin up a fresh `testing.postgresql` per test and call
  `prepare_database`, so tests never exercise an "upgrade existing DB" path.

## Option A — lightweight, recommended now (no Alembic)

Formalize what already exists.

1. Add `ord_schema/orm/migrations/` holding numbered, **idempotent** SQL files
   (`CREATE INDEX CONCURRENTLY IF NOT EXISTS ...`, `ALTER TABLE ... ADD COLUMN
   IF NOT EXISTS ...`). The partial-index SQL already in the README is migration
   `0001`.
2. Add a tiny `apply_migrations(engine)` helper (or a `psql -f` runbook) and
   document it in [orm/README.md](ord_schema/orm/README.md) alongside the
   existing "rebuild from ord-data" path as the two supported upgrade routes.
3. Keep `create_all` for new databases (it stays the bootstrap for fresh DBs).

Cost: ~an afternoon. No new runtime dependency, no second source of truth, no
enum/cartridge headaches. Closes the additive-change gap for existing DBs.

## Option B — adopt Alembic (the requested plan)

Phased so it can be paused after any phase.

### Phase 1 — scaffold
- Add `alembic` to the `tests`/runtime deps in `pyproject.toml`; `alembic init`.
- In `env.py`, set `target_metadata = Base.metadata` by importing
  `ord_schema.orm.mappers` (this triggers `build_mappers()`), so autogenerate
  diffs against the proto-generated schema.
- Configure: `include_schemas=True`, `version_table_schema="ord"`,
  `compare_type=True`, and an `include_object` hook that **ignores the `rdkit`
  schema and cartridge-owned objects** (Alembic must not try to manage them).

### Phase 2 — bootstrap & extensions (autogenerate can't see these)
- Keep `CREATE SCHEMA ord/rdkit` and `CREATE EXTENSION rdkit/tsm_system_rows/
  btree_gist` in the **first migration** (or a pre-upgrade step), since
  autogenerate never emits schema/extension/cartridge DDL. Decide explicitly
  that the migration owns `ord`/extensions and the cartridge owns `rdkit`.

### Phase 3 — baseline existing databases
- Generate the baseline migration representing the **current** schema.
- `alembic stamp head` on existing DBs so they are marked current and **not
  rebuilt or diffed**. This is what makes adoption non-disruptive.

### Phase 4 — switch the entry point
- Replace `Base.metadata.create_all(engine)` in `prepare_database()` with
  `alembic upgrade head` (programmatic `command.upgrade`). Either keep
  `create_all` for the test fixture or — better — run migrations in tests so CI
  actually exercises them.

### Phase 5 — enum strategy (Alembic's weak spot here)
- Proto enum edits are common and PG enums are painful: `ALTER TYPE ... ADD
  VALUE` can't run inside a transaction, and removals/renames need manual
  rewrites. Document that **enum changes are always hand-written migrations**,
  never trusted to autogenerate, with `op.execute(...)` + non-transactional
  migration where needed.

### Phase 6 — partial / GiST / CONCURRENTLY indexes
- Hand-write these: autogenerate emits the index but not `CONCURRENTLY`. Use
  `op.create_index(..., postgresql_concurrently=True, postgresql_where=...)` in
  a migration flagged non-transactional.

### Phase 7 — CI: test migrations *and* enforce sync with proto
- In CI (against `testing.postgresql`): run `alembic upgrade head` from an empty
  DB and from the baseline; optionally a downgrade round-trip.
- **Crucial sync check:** fail CI if the proto-generated `Base.metadata`
  diverges from the migrations head (`alembic check` / autogenerate-produces-
  empty). This is what keeps the two sources of truth aligned: when someone
  edits a `.proto`, the check forces them to add a migration.

### Phase 8 — docs
- Replace the manual-backfill section of [orm/README.md](ord_schema/orm/README.md)
  with `alembic upgrade head`; document the enum/cartridge rules above.

---

## The crux: two sources of truth

After Phase 4 the schema is defined **twice** — by `build_mappers()` (from proto)
and by the migration history. Phase 7's CI sync check is non-negotiable; without
it the two silently drift and `prepare_database` produces DBs that don't match
the models. This ongoing coupling is the main reason to defer until a trigger
justifies it.

## Risks / gotchas

- **Enum churn** (above) — the most likely source of broken migrations.
- **Cartridge ownership** — autogenerate will try to drop `rdkit`-schema objects
  it doesn't know about unless `include_object` excludes them.
- **Autogenerate noise** — dynamic, descriptor-derived names can produce noisy or
  unstable diffs; every migration needs human review.
- **Test realism** — today's per-test fresh DB never upgrades; without Phase 7
  the upgrade path stays untested.

## Decision

Do **Option A** now (closes the real gap cheaply). Revisit **Option B** when a
long-lived production DB, multiple deployments, or destructive schema changes
make a versioned migration history worth its permanent upkeep.
